# 协作与改造提交流程

这份文档用于约束本项目后续的协作改造方式，目标是保证：

- 代码可持续接手
- 文档和代码保持同步
- 换电脑后可以快速恢复上下文
- 每次提交都能看出“这次改了什么、还剩什么”

## 1. 开始改造前先做什么

每次开始开发前，先执行下面动作：

1. 拉取最新代码：

```bash
git checkout main
git pull origin main
```

2. 优先阅读这些文档：

- `docs/改造进度同步记录.md`
- `项目总说明.md`
- `docs/代码改造指南.md`

3. 确认本次要改的模块目录和影响范围。

## 2. 每次改造必须同步更新什么

只要改动属于“功能新增、能力升级、结构调整、部署变化、接口变化、文档变化”中的任意一种，就必须同步更新文档。

至少更新下面其中一项，通常建议一起更新：

- `docs/改造进度同步记录.md`
- `项目总说明.md`
- 模块自己的 `README.md`

## 3. 改造进度文档必须补哪些内容

更新 `docs/改造进度同步记录.md` 时，至少补充下面信息：

- 修改日期
- 本次改造主题
- 已完成内容
- 受影响模块
- 是否影响部署
- 是否影响接口
- 是否还有未完成部分
- 下一步建议

如果本次没有新增功能，只是修复问题，也要补一句“修了什么问题”。

## 4. 推荐的进度追加格式

建议直接在 `docs/改造进度同步记录.md` 末尾追加一段，格式如下：

```md
## 2026-07-09 补充：本次改造主题

本次完成：

- 
- 

影响范围：

- frontend:
- backend:
- worker:
- deploy:

验证方式：

- 

未完成项：

- 

下一步建议：

- 
```

## 5. 提交代码前检查清单

提交前至少检查下面几项：

- `git status` 确认没有误提交 `.env`、私钥、数据库文件、截图产物
- 受影响模块的说明文档是否一起更新
- `docs/改造进度同步记录.md` 是否补了这次改造记录
- 如果改了接口，是否同步更新 Swagger 说明或接口文档
- 如果改了部署方式，是否同步更新部署文档

## 6. 提交信息建议

提交信息尽量直接写能力名称，避免使用模糊描述。

推荐示例：

- `Add live UI execution viewer`
- `Add JMeter runner integration`
- `Add mock response service`
- `Fix token expiry redirect flow`
- `Update progress sync documentation`

## 7. 跨电脑继续开发的标准顺序

如果换到另一台电脑，建议按这个顺序恢复：

1. `git pull origin main`
2. 看 `docs/改造进度同步记录.md`
3. 看 `项目总说明.md`
4. 看本次要改模块下的 `README.md`
5. 启动项目后再开始改代码

## 8. 不要提交的内容

下面这些内容不要进入 Git：

- `.env`
- 私钥、公钥原文备份
- 数据库数据文件
- 临时附件
- 测试截图和录屏产物
- 本机缓存目录

## 9. 这份流程文档什么时候更新

如果后续团队协作方式、目录结构、部署方式、文档入口发生变化，就一起更新这份文档。
