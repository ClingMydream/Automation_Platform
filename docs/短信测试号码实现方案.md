# 短信测试号码实现方案

## 结论

电话号码本身不能通过随机算法“生成并获得收信能力”。收信能力来自运营商或云通信服务商对号码的实际分配、租用和路由配置。

本项目因此把号码分成三个边界清晰的模式：

1. 中国手机号格式：只测长度、前缀、表单和数据传递，不发送短信。
2. Twilio 测试凭据魔术号码：只模拟云通信 API 的成功或错误响应，不连接运营商，不会产生真实短信。
3. 自有/已租用接收号码：从 `TEST_SMS_PHONE_NUMBERS` 读取受控 E.164 号码，供真实端到端测试使用。

## 推荐落地方式

生产前的自动化测试建议分层：

- 单元和接口测试：Mock 短信供应商响应，覆盖成功、限流、失败、超时和回调验签。
- 集成测试：使用供应商测试凭据和魔术号码，验证请求格式和错误处理。
- 少量端到端测试：租用支持入站短信的号码，或使用团队自有测试 SIM 卡；将收到的短信通过供应商 webhook 或设备网关写入专用测试收件箱。

当前项目已经完成前两类数据入口，并为第三类提供受控号码池。若要在平台中直接查看验证码，下一步应新增：

1. `sms_test_messages` 存储表，设置较短保留期并对短信内容脱敏。
2. 供应商入站 webhook，校验供应商签名后写入收件箱。
3. 业务测试程序按号码和时间窗口轮询收件箱，而不是使用公共接码网站。
4. 号码租约、并发占用、自动释放和审计记录。

## Twilio 官方能力依据

- [Test Credentials](https://www.twilio.com/docs/iam/test-credentials)：测试凭据不会连接真实号码；魔术号码可模拟购号和发信 API，但不会真实投递。
- [Phone Numbers](https://www.twilio.com/docs/phone-numbers)：可搜索、购买和管理云号码；具体短信能力和合规要求取决于国家、号码类型及供应商库存。
- [Messaging Webhooks](https://www.twilio.com/docs/usage/webhooks/messaging-webhooks)：支持在号码收到短信时把入站消息回调到应用配置的 webhook。

## 不采用公共接码平台的原因

公共接码号码常被多人共享，短信可能被他人看到，号码也可能被目标系统封禁或随时失效。它不适合账号、验证码、个人信息或任何需要可重复性的自动化测试。
